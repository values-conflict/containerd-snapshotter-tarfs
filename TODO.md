- eliminate the overlayfs dependency in Prepare: instead of returning `type: overlay` with the FUSE dir as lowerdir, have the FUSE daemon itself serve a merged copy-on-write view (lower layers read-only from tar, upper layer written to the on-disk upperDir) and return a plain bind mount -- this removes the kernel overlayfs module from the picture entirely, sidestepping the CAP_EXPORT_SUPPORT / dentry-reconstruction issue that currently requires infinite FUSE entry/attr timeouts as a workaround; doing this will also require ditching go2fuse (whose `io/fs.FS` abstraction is read-only by design) and writing FUSE nodes directly against go-fuse
- zstd-compressed layers (produced by BuildKit) are always fully decompressed into a bytes.Buffer; standard single-frame zstd has no gsip-equivalent because zstd blocks share a window buffer and aren't independently seekable -- the pragmatic fix is to decompress once to a temp file and serve via an os.File ReadAt (file stays in the kernel page cache and doesn't hold RAM indefinitely); true random access would require the zstd:chunked format (independent frames + TOC), which BuildKit doesn't produce
- extend toward lazy pull (opt-in, eg via a `--lazy` flag or snapshotter label): instead of fetching the full compressed blob before mounting, stream layer data on-demand via HTTP range requests against the registry -- containers would start before the pull completes, analogous to SOCI/eStargz but without requiring a specially-built image format; default behavior stays fetch-then-mount to keep the happy path simple
