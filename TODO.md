- eliminate the overlayfs dependency in Prepare: instead of returning `type: overlay` with the FUSE dir as lowerdir, have the FUSE daemon itself serve a merged copy-on-write view (lower layers read-only from tar, upper layer written to the on-disk upperDir) and return a plain bind mount -- this removes the kernel overlayfs module from the picture entirely, sidestepping the CAP_EXPORT_SUPPORT / dentry-reconstruction issue that currently requires infinite FUSE entry/attr timeouts as a workaround; doing this will also require ditching go2fuse (whose `io/fs.FS` abstraction is read-only by design) and writing FUSE nodes directly against go-fuse

- extend toward lazy pull (opt-in, eg via a `--lazy` flag or snapshotter label): instead of fetching the full compressed blob before mounting, stream layer data on-demand via HTTP range requests against the registry -- containers would start before the pull completes, analogous to SOCI/eStargz but without requiring a specially-built image format; default behavior stays fetch-then-mount to keep the happy path simple
  - this also surfaces a partial-verification gap: HTTP range requests return a slice of the compressed blob, but the outer content-store digest can't be verified until the whole blob arrives -- meaning the filesystem would serve structurally unverified bytes during the in-flight window; BAO (Blake3 Verified Streaming, https://github.com/oconnor663/bao) addresses exactly this shape of problem: it encodes a Merkle tree over the blob so any range-fetched chunk can be immediately verified against the root hash without having seen the rest of the blob; for this to work at OCI scale the registry would need to store and serve the BAO hash tree alongside each blob (as a referrers-based side-car descriptor), and the image format would need a convention for advertising the BAO root -- no such standard exists today; this seems worth pushing for as part of any lazy-pull standards work, with eStargz/SOCI's per-entry digest table as prior art (they solve the same problem at tar-entry granularity by embedding a digest index in the image, but require a specially-built image format -- BAO would work on any blob without a re-pack)
  - a related angle: explore combining the BAO hash tree and a compressed-blob seek index (zstd frame-boundary table, or a gzip equivalent) into a single side-car format -- the two structures are complementary (seek index: where does frame N start and what decompressed range does it cover; BAO tree: what is the verifiable hash of each compressed chunk), and when zstd frame sizes align with BAO chunk sizes the two tables are in 1:1 correspondence, making a unified descriptor natural; a combined format -- something like a list of `(compressed_offset, decompressed_offset, bao_leaf_hash)` triples plus a BAO root -- would let lazy pull fetch one side-car blob and get both random-access decompression and per-chunk integrity in one shot; no standard for this exists; worth exploring what alignment constraints a practical spec would impose on image builders (eg, fixed zstd frame sizes at build time) and whether the result is proposable to the OCI working group

- persist the gsip checkpoint index in the content store so the sequential scan cost is paid once per blob (surviving process restarts) rather than on every `openBlobAsFS` call: `gsip.Reader.Encode` and `gsip.Decode` already exist in the library -- the write path is to call `r.Encode(&buf)` after `tarfs.New` completes (with care that the checkpoint drain goroutine has finished -- the current `runtime.Gosched()` is not a guarantee; gsip may need a `Wait()` or similar), ingest the JSON bytes as a new content blob, then label the source gzip blob `containerd.io/gc.ref.content.gsip-index = <index-digest>`; the read path checks for that label first and calls `gsip.Decode(ra, size, indexReader)` if found, falling back to `gsip.NewReader` on any failure; the GC lifecycle comes for free -- when the source blob is collected, nothing else holds a reference to the index blob, so it is collected in the same GC pass; size caveat: each checkpoint carries a 32 KiB `Hist` window (the deflate sliding window), base64-encoded in JSON, so the index runs roughly 5-10% of the compressed blob size for typical gzip layers -- a binary encoding would be more space-efficient, but the existing JSON path works today; stability caveat: `gsip.Index` carries an explicit "the layout will absolutely change" warning, so decode failures must always fall through to `gsip.NewReader` (never hard-fail); this optimization only applies to gzip blobs ≥ 4 MiB (smaller blobs already use the `bytes.Buffer` fast path and never touch gsip)

- investigate a gsip-equivalent for zstd: unlike gzip's deflate blocks (which share a 32 KiB sliding-window history between blocks), zstd frames are independently decompressible -- a seek index only needs `(compressed_frame_start, decompressed_frame_start)` pairs with no window payload, so the index for a typical layer is a few kilobytes vs gsip's ~5-10% overhead; the write path scans the compressed blob once to collect frame-boundary offsets, ingests them as a small content-store blob, and labels the source blob with a pointer (same lifecycle/GC pattern as the proposed gsip-index TODO); the read path checks the label first, locates the nearest frame boundary at or below the requested decompressed offset, and decompresses only the frames needed, falling back to `ingestDecompressedBlob` on any failure; `klauspost/compress/zstd` does not expose frame-seeking directly, so the boundary scan must implement enough of the zstd framing spec to hop over frames without fully decoding them (the frame header format is simple and publicly documented); separately, the seekable-zstd extension (independently-compressed frames plus an appended seek table) would make this trivial if the image builder emits it -- worth checking whether BuildKit can be configured to produce seekable-zstd layers, but the frame-scan path is required as a fallback regardless
